Tiramisu
According to IR, DL compilers can be classified to 2 groups + 1, the 1st group use computation graph as their IR and
mainly focus on graph level optimization, the 2nd group use more low level “Tensor Loop Algebra” as their IR and
mainly focus on operator level optimization, TVM can belong to each of them, because it has all these two level IR
and optimization, the MLIR doesn’t belong to neither of them, because it isn’t a concrete DL compiler, it is a compiler
infrastructure.
This slides mainly focus on the 2nd group, so the 1st group and MLIR will be introduced briefly.

Solutions to Simplify High Performance Optimization
Halide
Authors: Jonathan Ragan-Kelley, Andrew Adams @2012.
Theoretical Basis: Interval Analysis.
Decouple algorithm from the optimization.
❖ Users must use Halide way to represent their algorithm,
in Halide this part is called “algorithm” .
❖ (Optional) Users can use Halide way to represent the
optimization sequence they want to do, in Halide this
part is called “schedule”.
Polyhedral
Authors: Lots of Researchers from 1960s.
Theoretical Basis: Parametric Polyhedral, Presburger Relations
and so on.
Abstract the high performance optimization
problem to a mathematical form, and solve it
through integer linear programming(ILP).
Definination from This Paper
多面体编译技术是指在循环边界约束条件下将语句实例表示
成空间多面体,并通过这些多面体上的几何
操作来分析和优化程序的编译技术,这种模型称为多面体模型.

Polyhedral Schedule Representation
https://www.slideshare.net/ahayashi10/introduction-to-polyhedral-compilation
➢ Each statement have its own schedule.
➢ Schedule of different statement must can
represent the execution order between
them, the execution order is represented by
lexicographical order.
➢ For example, given the schedules of these 3
pictures, we can know statement S1 must be
executed before S2, and all of instances of
S2 must be executed before any instance of
S3.
Polyhedral Transformations with Schedules
https://www.csa.iisc.ac.in/~udayb/slides/uday-polyhedral-opt.pdf
➢ The transformation is method to
achieve some goals.
➢ Improving parallelism, locality are
two common goals.
➢ For parallelism, if the points which
don’t dependent on each other are
parallel with a axis, then they can
be computed in parallel at that axis.
So we need find the transformation
that can make the points parallel
with axis.
➢ The key of transformation is get the
corresponding schedule.
Original Schedule: T(i, j) = (i, j)
New Schedule: T(i, j) = (i + j, j)
Transformation: (i, j) → (i + j, j)

Polyhedral Conclusion
Review The Three-Stage Process
1. Program analysis
❖ Extract polyhedral representation of iteration domains.
❖ Extract dependences.
2. Optimization
❖ Compute schedules through scheduling algorithm like “Pluto”.
3. Code generation
❖ “Apply” the transformation(from schedules) in the polyhedral
representation.
❖ Scanning points in the transformed space in lexicographically
increasing order(from schedules).
Tips
✓ All of these three stages can be done automatically through tools.
✓ The schedule computing process is done iteratively.
✓ The code generation still need solve ILP problem.

TC Code of Matrix-Vector Production
➢ User only need provide “Compute/Algorithm”, the
“Schedule/Optimization” will be got automatically .
➢ Notation borrows from the Einstein notation (a.k.a.
summation convention).
➢ A and x are input tensors and C are output.
➢ Variables ‘i’ and ‘k’ aren’t defined anywhere,
implicitly become index variables.
➢ The ranges of ‘i’ and ‘k’ are inferred as i = [0,M), and
k = [0,K).
➢ Because k only appears on the right-hand side,
stores into C will reduce over k with the reduction
operator +.

Weights
The scales are directly calculated based on the value of the
weights.
➢ Support power2 and max modes.
➢ Both modes find the maximum value within the weight tensor first.
➢ In power2 mode, the maximum is rounded down to power of two.
➢ In max mode, the maximum is used as the scale. Without rounding,
max mode might have better accuracy in some cases.
Intermediate Feature Maps
The scales can be found with data-aware quantization method
and pre-defined global scales is supported.
➢ Data-aware quantization method takes a calibration dataset as the
input argument. Scales are calculated by minimizing the KL
divergence between distribution of activation before and after
quantization.
➢ Pre-defined global scales saves the time for calibration. But the
accuracy might be impacted.

TVM AutoTVM Schedule Explorer
https://tvm.apache.org/docs/tutorials/autotvm/tune_simple_template.html, https://arxiv.org/pdf/1802.04799.pdf
@autotvm.template("tutorial/matmul")
def matmul(N, L, M, dtype):
A = te.placeholder((N, L), name='A', dtype=dtype)
B = te.placeholder((L, M), name='B', dtype=dtype)
k = te.reduce_axis((0, L), name='k')
C = te.compute((N, M), lambda i, j:
te.sum(A[i, k] * B[k, j], axis=k),
name='C')
s = te.create_schedule(C.op)
# schedule
y, x = s[C].op.axis
k = s[C].op.reduce_axis[0]
##### define space begin #####
cfg = autotvm.get_config()
cfg.define_split("tile_y", y, num_outputs=2)
cfg.define_split("tile_x", x, num_outputs=2)
##### define space end #####
# schedule according to config
yo, yi = cfg["tile_y"].apply(s, C, y)
xo, xi = cfg["tile_x"].apply(s, C, x)
s[C].reorder(yo, xo, k, yi, xi)
return s, [A, B, C]
N, L, M = 512, 512, 512
task = autotvm.task.create("tutorial/matmul", args=(N, L, M, 'float32'), target='llvm')
Tuner Pseudo Code
ct = 0
while ct < max_number_of_trials:
propose a batch of configs
measure this batch of configs on hardware and get results
ct += batch_size
When proposing the next batch of configs, the tuner
can take 4 different strategies.
RandomTuner: Enumerate the space in a random order
GridSearchTuner: Enumerate the space in a grid search order
GATuner: Using genetic algorithm to search through the space
XGBTuner: Uses a model based method. Train a XGBoost
model to predict the speed of lowered IR and pick the next
batch according to the prediction.
